# Contact Trust Reconciliation and Guarded Remediation

PR #68 introduces a deterministic, provider-free reconciliation tool for historic contact trust debt. It does not call Apollo, Hunter, LinkedIn, an LLM, Projectory or any other provider.

## Safety model

The tool is read-only unless all of the following are supplied:

```text
--apply
--manifest <sealed-manifest.json>
--confirm-hash <manifest-sha256>
```

Generation is the default mode. Apply is limited to 25 approved rows unless the operator supplies an explicit `--max-apply` value.

A sealed manifest records a complete fingerprint of the relevant contact, link, project and evidence tables. Apply stops if the database snapshot or an approved contact record changed after generation. A second application of an already-applied manifest exits as an idempotent no-op.

## Evidence precedence

The reconciliation policy uses this order:

```text
human email verification
→ valid Hunter mailbox evidence
→ verified Apollo contact state
→ unverified source data
→ historic generated pattern
```

Hunter `accept_all` is not mailbox verification. A provider- or human-verified email is retained even when its formatting happens to match the old deterministic pattern.

## Exact historic generated-email fingerprint

Only the former application algorithm is reproduced:

1. lower-case the contact name;
2. take the first and last name components;
3. lower-case the company;
4. remove `pty`, `ltd`, `limited`, `inc`, `corp`, `group`, `australia`, `holdings`;
5. remove spaces and non-alphanumeric characters;
6. build `first.last@cleanedcompany.com.au`.

Other common corporate formats are never classified as application-generated by this tool.

## Dispositions

Every contact receives one disposition:

```text
safe_keep
safe_demote
safe_promote
safe_clear_generated_email
safe_link_to_project
manual_review
no_change
```

Only the four safe correction dispositions can be approved for apply:

```text
safe_demote
safe_promote
safe_clear_generated_email
safe_link_to_project
```

Duplicate identity candidates are always routed to `manual_review`. Same name plus employer is a review signal only; it never triggers an automatic merge.

An orphan can be linked automatically only when its legacy project text matches exactly one non-merged project name.

## Workflow

### 1. Generate a dry-run manifest

```bash
pnpm exec tsx server/scripts/contactTrustReconcile.ts \
  --output-dir ./artifacts/contact-trust
```

Outputs:

```text
contact-trust-manifest.draft.json
contact-trust-manifest.csv
contact-trust-summary.json
```

All rows are generated with `approved: false`.

### 2. Review

Change only `approved` from `false` to `true` for reviewed safe rows. Do not edit the disposition, evidence, expected state or record hash.

### 3. Seal

```bash
pnpm exec tsx server/scripts/contactTrustReconcile.ts \
  --seal \
  --manifest ./artifacts/contact-trust/contact-trust-manifest.draft.json \
  --output-dir ./artifacts/contact-trust
```

Sealing rejects any change other than the `approved` flag and returns the SHA-256 confirmation hash.

### 4. Canary apply

```bash
pnpm exec tsx server/scripts/contactTrustReconcile.ts \
  --apply \
  --manifest ./artifacts/contact-trust/contact-trust-manifest.sealed.json \
  --confirm-hash <sha256> \
  --project-ids <current-this-week-project-ids> \
  --max-apply 10 \
  --output-dir ./artifacts/contact-trust/canary
```

Optional filters:

```text
--contact-ids 1,2,3
--project-ids 101,102
--dispositions safe_demote,safe_promote
--max-apply 10
```

Outputs:

```text
contact-trust-apply-before.csv
contact-trust-apply-after.csv
contact-trust-apply-summary.json
```

## Apply behaviour

- `safe_demote`: moves the contact to `named_unverified` and clears stale verification flags; the email is retained unless it is an exact unsupported generated pattern.
- `safe_promote`: requires conclusive provider or human evidence and sets `emailVerified=true`, `verificationStatus=verified`, `contactTrustTier=send_ready`.
- `safe_clear_generated_email`: clears only an exact historic pattern without later valid evidence and demotes to `named_unverified`.
- `safe_link_to_project`: inserts one missing `contactProjects` row after an exact single-project match.

No contact is deleted or merged. No provider is called. No email is sent. No project, account, Full Potential, pipeline or C4C record is changed.

## Recommended production sequence

1. Run full dry-run generation.
2. Review current This Week and Next Best 5 contacts first.
3. Approve no more than ten contacts for the first canary.
4. Create a production safety checkpoint.
5. Apply the sealed canary manifest.
6. Verify contact counts remain unchanged, link counts change only for approved link rows, and This Week/Next Best 5 project ordering is unchanged.
7. Expand only after the canary evidence matches the manifest exactly.
